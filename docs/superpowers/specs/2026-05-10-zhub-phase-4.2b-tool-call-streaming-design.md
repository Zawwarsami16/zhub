# Phase 4.2b — True chunked tool_call streaming through SSE

**Date:** 2026-05-10
**Status:** spec
**Goal:** complete OpenAI-compatibility for streamed tool calls. Today (Phase 4.2) the hub supports `X-Zhub-Stream-Tools: pre-resolve` which buffers the full publisher response, runs the tool-call resolution dance, then emits the final text as one SSE chunk. After 4.2b the hub streams tool_call argument deltas through SSE *as they arrive* from the publisher, matching what Cursor / Continue / native OpenAI SDK clients expect.

---

## Why

Modern OpenAI-compat dev clients (Cursor agentic mode, Continue, the official `openai` SDK with `stream=True`) parse incremental SSE chunks of the form:

```
data: {"choices":[{"delta":{"tool_calls":[{"index":0,"id":"call_x","type":"function","function":{"name":"get_weather","arguments":""}}]}}]}
data: {"choices":[{"delta":{"tool_calls":[{"index":0,"function":{"arguments":"{\"city\""}}]}}]}
data: {"choices":[{"delta":{"tool_calls":[{"index":0,"function":{"arguments":":\"NYC\"}"}}]}}]}
data: {"choices":[{"delta":{},"finish_reason":"tool_calls"}]}
```

zhub today drops these — only `delta.content` text strings flow through. After 4.2b they pass through the full chain (brain adapter → publisher SDK → hub WS → SSE) so any standard OpenAI client just works.

## Modes

The Phase 4.2 `X-Zhub-Stream-Tools` header gains two more values:

| `X-Zhub-Stream-Tools` value | Hub behavior |
|---|---|
| (unset) — **default** | Pass tool_call deltas through verbatim as SSE chunks. The client receives them as standard OpenAI streaming. Hub does NOT auto-resolve. |
| `pre-resolve` | (Phase 4.2 — unchanged) Buffer everything, run non-streaming auto-resolve loop, emit final resolved text as one SSE chunk. |
| `auto` | **NEW.** Pass tool_call deltas through. When `finish_reason: tool_calls` arrives, hub auto-resolves in parallel against connected capabilities + exposures, then issues a follow-up streaming chat-request to the publisher with `role: tool` messages appended. The follow-up's chunks continue the same SSE stream — client sees tool_call deltas, then a synthetic `role: tool` SSE event with the result, then the publisher's final answer text. |

(`X-Zhub-Tool-Resolve: client` is the legacy non-streaming opt-out and remains unchanged.)

## Non-goals

- Tool-call streaming for the brain adapters that don't natively emit tool_calls (Ollama Llama 3.2 3B doesn't have function-call wire support; the adapter just won't emit them).
- Mid-stream cancellation. Once a tool_call deltas burst starts, it completes.
- Tool retry policies (separate Phase 7.x).

## Architecture

Three layers change, in order:

1. **Brain adapters** — currently `ChatChunk(delta=str, done, finish_reason, raw)`. Add an optional `tool_call_delta: dict` field. When the upstream SSE includes a `tool_calls` delta, surface it in this field instead of (or in addition to) `delta`. Each adapter parses its native shape.

2. **Publisher SDK (`zhub.client._handle_chat`)** — currently yields strings → emits `chat-chunk` envelopes with `delta: str`. Generalize: yielded items can be strings (today's path) OR dicts. Dicts may carry `delta`, `tool_call_delta`, `done`, `finish_reason`. The SDK forwards the shape through `chat-chunk` payload verbatim.

3. **Hub (streaming chat path)** — read chat-chunks from publisher's queue:
   - `delta` field → forward as today (text delta SSE chunk)
   - `tool_call_delta` field → forward as OpenAI-shape `delta.tool_calls` SSE chunk
   - On `done=true` with `finish_reason="tool_calls"`:
     - If mode == `auto`: stop SSE temporarily, accumulate the tool_calls (concatenate function.arguments across all deltas), run parallel resolve, append `role: tool` messages, issue new streaming proxy_chat, continue the SSE stream with the new chunks
     - If mode unset (default): emit final SSE chunk with finish_reason and DONE marker. Client handles tool_calls itself.

## New wire shape on the WS chat-chunk envelope

```json
{
  "type": "chat-chunk",
  "request_id": "...",
  "payload": {
    "delta": "text fragment OR empty when this is a tool_call chunk",
    "tool_call_delta": {                         // NEW
      "index": 0,
      "id": "call_abc",
      "type": "function",
      "function": {"name": "get_weather", "arguments": "{\"ci"}
    },
    "done": false,
    "finish_reason": null
  }
}
```

Both fields are optional. A chunk has either `delta` text or a `tool_call_delta` object — never both.

## Brain adapter changes

Each adapter that wraps an OpenAI-compat upstream (`groq`, `openai`, `cerebras`) parses `delta.tool_calls[]` from the SSE payload and surfaces as `ChatChunk(tool_call_delta={...})`. Anthropic uses a different shape (`content_block_delta.input_json_delta`); the adapter normalizes. Ollama: skip — no native function-call wire.

```python
@dataclass
class ChatChunk:
    delta: str = ""
    done: bool = False
    finish_reason: Optional[str] = None
    tool_call_delta: Optional[dict] = None    # NEW
    raw: Optional[dict] = None
```

## Publisher SDK changes

Today (`_handle_chat` in `zhub/client.py`):
```python
async for chunk in result:                          # result is async-gen of strings
    await ws.send(chat_chunk(str(chunk), env.request_id).to_json())
```

After:
```python
async for chunk in result:
    if isinstance(chunk, str):
        await ws.send(chat_chunk(delta=chunk, request_id=env.request_id).to_json())
    elif isinstance(chunk, dict):
        # operator yielded structured chunk (text, tool_call delta, finish, ...)
        await ws.send(Envelope(
            type="chat-chunk",
            request_id=env.request_id,
            payload=chunk,
        ).to_json())
```

Brain adapters yield `ChatChunk` objects. Publisher's chat_handler that wraps an adapter just forwards them. The SDK serializes ChatChunk-shaped dicts naturally.

## Hub changes

In `chat_completions` streaming branch:

```python
mode = request.headers.get("x-zhub-stream-tools", "").lower()
# default behavior (mode == ""): pass tool_call deltas through, no resolve
# "pre-resolve": existing Phase 4.2 path
# "auto": pass through + auto-resolve and continue stream when finish_reason is tool_calls

if mode == "pre-resolve":
    return _preresolved_stream_response(...)  # existing

# default + auto path: stream loop with optional resolve continuation
async def event_stream():
    accumulated_tool_calls: dict[int, dict] = {}  # by index
    while True:
        chunk = await queue.get()
        if chunk is None:
            break
        if chunk.get("tool_call_delta"):
            tcd = chunk["tool_call_delta"]
            idx = tcd.get("index", 0)
            # accumulate for potential auto-resolve later
            slot = accumulated_tool_calls.setdefault(idx, {"function": {"arguments": ""}})
            if "id" in tcd:
                slot["id"] = tcd["id"]
            if "type" in tcd:
                slot["type"] = tcd["type"]
            fn_delta = tcd.get("function", {})
            if "name" in fn_delta:
                slot.setdefault("function", {})["name"] = fn_delta["name"]
            if "arguments" in fn_delta:
                slot.setdefault("function", {}).setdefault("arguments", "")
                slot["function"]["arguments"] += fn_delta["arguments"]
            # forward as SSE delta.tool_calls chunk
            yield _format_tool_call_sse_chunk(tcd, completion_id, model, created)
            continue
        if chunk.get("done"):
            finish = chunk.get("finish_reason", "stop")
            if finish == "tool_calls" and mode == "auto" and accumulated_tool_calls:
                # auto-resolve and continue stream
                tool_calls = [accumulated_tool_calls[i] for i in sorted(accumulated_tool_calls)]
                # synthesize role:assistant + role:tool messages, send next chat-request
                async for next_chunk in _resolve_and_continue_stream(
                    ai_name, messages, tool_calls,
                    model, temperature, max_tokens, tools, tool_choice,
                ):
                    yield next_chunk
                # the resolve-and-continue handles its own done + DONE
                return
            yield _format_done_sse(finish, completion_id, model, created)
            yield "data: [DONE]\n\n"
            break
        # text delta
        yield _format_text_delta_sse(chunk.get("delta", ""), completion_id, model, created)
```

`_resolve_and_continue_stream` runs the same parallel-resolve logic as the non-streaming path, then opens a new `proxy_chat(stream=True)` against the publisher with the appended messages, and forwards its chunks (recursing through this same logic in case THAT response also emits tool_calls).

## Tests

`tests/test_tool_call_streaming.py`:

1. **`default_mode_passes_tool_call_deltas_through`** — publisher emits two tool_call delta chunks then finish_reason:tool_calls. Default mode SSE response includes both `delta.tool_calls` chunks and a final `finish_reason:tool_calls` chunk + `[DONE]`. Hub does NOT auto-resolve.

2. **`auto_mode_resolves_and_continues_stream`** — publisher emits tool_call deltas, finish_reason:tool_calls, then on the follow-up emits text "ok done". Connected capability fires once. Final SSE has tool_call delta chunks, then text delta chunks "ok done", then finish_reason:stop.

3. **`pre_resolve_mode_unchanged`** — sanity check Phase 4.2 path still works.

4. **`brain_adapter_groq_emits_tool_call_delta`** — mock SSE from groq with tool_call delta → adapter yields ChatChunk with tool_call_delta populated.

5. **`brain_adapter_anthropic_emits_tool_call_delta`** — mock anthropic input_json_delta event → adapter normalizes.

6. **`publisher_sdk_forwards_dict_chunks`** — chat_handler yields dict with tool_call_delta → publisher emits chat-chunk with the same payload.

## Out of scope

- Mid-stream tool resolution cancellation (client disconnects mid-resolve)
- Tool retry on transient invoke errors (Phase 7.x)
- Sub-second cap on resolution roundtrip count (already bounded by max_iters in non-streaming path; reuse here)
- Tool_calls during non-streaming chat-chunk (publishers using sync chat_handler can't emit them — only async-gen path supports tool_call_delta)
