## Phase 1.5 — Auto-reconnect on publisher and connect

### Problem

Hand-test surfaced: when the hub restarts (or any WebSocket drop happens — network blip, hub crash, transient error), both `publish()` and `connect()` runners exit permanently. The async task completes; no retry. Operator must re-call `publish(...)` manually. This is a production-readiness gap.

### Approach

Wrap the WebSocket session in an **outer reconnect loop** with **exponential backoff**:

- Initial delay: 1s
- Backoff multiplier: 2x
- Max delay: 60s
- Reset on successful registration

On reconnect, re-send the register envelope. Publishers automatically include the stored `api_key` so key pinning re-attaches them to the same name (this already works — see Phase 1.0a key pinning logic on the server).

### Behavior

```
[publish/connect call] → [outer loop] → [open WS] → [register] → [serve until disconnect]
                            ↑                                              ↓
                            └────[backoff sleep]────────────[disconnect]──┘
```

When `pub.api_key` is already set from a previous registration, the next reconnect sends `register-publisher` with that key — server's key-pinning re-register path accepts it. No new key allocated.

### Out of scope

- **Connection-state callback to chat handler:** if a chat request arrives mid-disconnect, it fails — caller retries. Don't queue requests across disconnects in this phase.
- **Heartbeat ping during quiet times:** WS will detect dead connections eventually (on next send/receive). Active heartbeat is Phase 1.5b if needed.

### TDD plan

`tests/test_reconnect.py`:
1. Start hub, register publisher, verify api_key set.
2. Stop the hub thread.
3. Wait briefly.
4. Start a fresh hub on the same port using the same db_path.
5. Send a chat request after the new hub is up — verify response (proves publisher re-registered).
6. Same flow for connect-side.

## Phase 1.6 — Pocket ↔ zhub native integration

### Context

**Pocket** is Father's browser-only AI chat SPA at `github.com/Zawwarsami16/pocket`. Vite + React + TypeScript. Browser-direct API calls. Already supports a "Custom OpenAI-compatible" provider, so technically a zhub publisher works today if the user manually pastes `<hub-url>/<name>/v1` into the base URL field.

### Goal

Make zhub a **first-class** Pocket provider. Three value-adds beyond raw Custom-provider:

1. **Auto-detect** zhub URL shape (`*/<name>/v1` or `*/<name>/manifest.json`) and offer to add as a "zhub" provider with proper labeling.
2. **Manifest preview** in Settings — when a zhub URL is configured, fetch `/<name>/manifest.json` and show: name, description, operator, capabilities list, signed/unsigned status, public_key (truncated), connection count.
3. **Live connection awareness** (later, optional) — open a SHARED WebSocket from Pocket to the hub via `/ws/connect`, get connection events. For Phase 1.6 the static manifest fetch is enough; live updates are 1.6b.

### Architecture

```
Pocket SPA (browser)
   ├── src/providers/zhub.ts     ← new provider
   ├── src/providers/registry.ts ← register zhub
   └── src/ui/Settings.tsx       ← surface manifest preview when zhub provider selected
```

The zhub provider is essentially the existing OpenAI-compat provider with three additions:

- `validate(base_url, key)`: HEAD `/manifest.json` to confirm hub responds
- `fetchManifest(base_url)`: GET `/<name>/manifest.json`, return parsed manifest
- `cors_status: "varies"` (depends on the hub's CORS config — most allow it)

The chat path is unchanged from OpenAI-compat — Pocket's existing infrastructure handles streaming, errors, etc.

### Out of scope

- Pocket-as-zhub-client (browser exposing capabilities back via WebSocket from `/ws/connect`). Browser WS to zhub works in principle but the value is unclear without a concrete capability — defer to operator demand.
- Signature verification client-side. Pocket displays signed status from the manifest (the publisher's claim) but doesn't independently verify the ed25519 signature — that needs a JS ed25519 library. Phase 1.6b.

### Working environment caveat

Local Termux can't run `npm run build` (rollup native binding blocked per Pocket's project memory). Verification is by:

1. TypeScript compile (`tsc --noEmit`) — catches type errors locally.
2. Vercel preview deploy after push — catches build errors.
3. Manual browser test against a real zhub hub.

### TDD plan

For TS code without a runnable test harness in proot, the discipline becomes:

1. Write the new provider with strict types.
2. Run `tsc --noEmit` — catches type drift.
3. Lint with project's existing setup.
4. Commit + let Vercel build verify.

I'll add a minimal Vitest test for the URL-shape detection (`detectZhubUrl()` is pure logic, easy to test without DOM).

## Phase 1.3 — MCP bridge (MCP server → zhub publisher)

### Context

Anthropic's [Model Context Protocol](https://modelcontextprotocol.io) is becoming the standard for AI tools/resources to expose themselves to LLM clients (Claude Desktop, Cline, etc.). MCP servers expose **tools** (function-calls), **resources** (read-only data), and **prompts** (templates). MCP wire format is JSON-RPC 2.0 over stdio (or SSE).

### Goal

Wrap any existing MCP server as a zhub publisher. The MCP tools become zhub-side **capabilities** that show up in the manifest. Chat requests sent to the wrapped publisher get translated into MCP tool calls.

### Approach

```
[zhub HUB]
   │
   │ chat-request envelope
   ▼
[mcp_bridge.py — a zhub publisher]
   │
   │ JSON-RPC: tools/list at startup, tools/call per chat
   ▼
[MCP server — runs as subprocess, stdio]
```

The bridge is a single Python script:

```python
# examples/mcp_bridge.py
from zhub import publish
from zhub.mcp import MCPClient

mcp = MCPClient(["uvx", "mcp-server-filesystem", "/some/path"])
await mcp.start()
tools = await mcp.list_tools()

async def chat_handler(messages, options):
    # Simplest: invoke the first tool with the user's text as input.
    # Real version: route based on message content or tool-call envelope.
    last = messages[-1]["content"]
    if tools:
        result = await mcp.call_tool(tools[0]["name"], {"input": last})
        return result
    return "no tools available"

pub = publish(
    name="mcp-fs",
    description="MCP filesystem server via zhub",
    chat_handler=chat_handler,
    capabilities=[Capability(name=t["name"], description=t.get("description", ""))
                  for t in tools],
    hub_url="ws://localhost:8080",
)
```

### What lands

- `zhub/mcp.py` — a tiny MCP client that speaks JSON-RPC over a subprocess's stdio. Three operations: `start()`, `list_tools()`, `call_tool(name, args)`.
- `examples/mcp_bridge.py` — a working bridge script using a real public MCP server.
- `tests/test_mcp.py` — a synthetic test using a stub stdio process that responds to JSON-RPC. No real MCP server dependency in CI.

### Out of scope (Phase 1.3b)

- Reverse direction (zhub publisher → MCP server) — MCP server endpoint exposing zhub publishers as MCP resources.
- MCP **resources** + **prompts** support. Tools first.
- Streaming MCP responses. tools/call is request/response in v0.

### TDD plan

`tests/test_mcp.py`:
1. Spin up a Python subprocess that responds to JSON-RPC over stdio (mock MCP server) — defined inline as a tiny Python script.
2. Wrap it with `MCPClient`. List tools. Call one. Assert response shape.
3. No reliance on any external MCP package or real MCP server.

Real MCP server integration (`uvx mcp-server-everything` etc.) is verified by the bridge example, manually run.
