## Phase 1.7 — Rate-limit enforcement

### Problem

Manifests advertise `rate_limit: "60/min"` today; the hub ignores it. Public endpoints can be hammered. Real public-deployment readiness needs server-side enforcement.

### Approach

In-memory sliding-window counter keyed by `api_key`. Lost on restart — that's fine for v0; clients that hit limits retry, no persistent damage.

Parse the same string format manifests use: `"60/min"`, `"10/s"`, `"1000/day"`. Default fallback: `"60/min"` if manifest doesn't specify.

Enforce on the HTTP `/v1/chat/completions` route (and `/registry/global`'s aggregator if useful — Phase 1.7b). When over limit, return HTTP 429 with `Retry-After: <seconds>` header, JSON body `{"error": {"code": "rate_limited", "message": "...", "retry_after": N}}`.

Public endpoints (no auth header): use a per-IP fallback window — generous default (`60/min`) so casual `curl /healthz` isn't blocked. Authenticated calls use the publisher's manifest rate.

### Out of scope

- Persistent rate-limit state across restart (Phase 1.7b)
- Per-connection-id rate limits (the publisher can decide)
- Distributed rate limits across federated hubs (Phase 1.7c — needs a shared store)

### TDD

`tests/test_rate_limit.py`:
1. Parse fixture: `"60/min"` → `(60, 60s)`, `"10/s"` → `(10, 1s)`, `"1000/day"` → `(1000, 86400s)`. Bad strings raise.
2. Sliding-window counter over a fake clock: 3 hits in window → allow; 4th hit at same instant → deny; window expires → next hit allows.
3. e2e: publish AI with `rate_limit: "3/s"` in manifest. Hit `/v1/chat/completions` 4 times rapidly. First 3 = 200, 4th = 429 with `Retry-After`.

## Phase 1.2 — JS/TS client library

### Goal

Browsers and Node services connect to zhub via a clean SDK without going through `openai` library or hand-coding WebSockets. Same primitives as Python: `publish()` and `connect()`.

### Layout

```
js/
  package.json        — name "@zawwarsami/zhub", workspaces nothing
  tsconfig.json       — strict, ESM-only target
  src/
    index.ts          — re-exports
    manifest.ts       — Manifest, Capability types
    protocol.ts       — Envelope + helpers (mirrors Python)
    client.ts         — publish(), connect(), ZhubPublication, ZhubConnection
    errors.ts
  test/
    protocol.test.ts  — envelope round-trip
    e2e.test.ts       — spin up the Python hub via fixture, run JS client against it
  README.md
```

### Behaviors

`publish(opts)`:
- `name`, `description`, `chatHandler` (sync function or async generator).
- Opens WebSocket. Auto-reconnect on disconnect (mirrors Phase 1.5).
- Returns `ZhubPublication` with `apiKey`, `name`, `baseUrl`, `listConnections()`, `findCapability()`, `invoke()`.

`connect(opts)`:
- `aiName`, `apiKey`, `capabilities` map (name → {schema, handler}).
- Returns `ZhubConnection` with `chat(messages)`, `chatStream(messages)` (async iterator), `close()`.

### Browser caveat

Browsers can't write raw TCP, but native `WebSocket` works. Browser builds therefore work for `connect()` mode (the most useful client-side direction). `publish()` from browser is theoretically supported (everything is WS) — useful for Pocket-as-publisher (a browser AI). Tests cover Node; browser path verified manually.

### Build

`tsc` produces ESM in `dist/`. Single bundle for browser is unnecessary at this stage — tree-shaken imports work via bundlers (Vite, esbuild, webpack).

### TDD

- `protocol.test.ts`: envelope JSON round-trips identical to Python's protocol.py output for at least register-publisher, chat-request, chat-response, invoke-request, invoke-result.
- `e2e.test.ts`: a Vitest suite that spawns `python -m zhub.server` as a subprocess, waits for `/healthz`, then runs the JS client against it. publish + connect + chat + invoke happy paths.

### CI

Add `js-test` job to `.github/workflows/ci.yml`. Uses `actions/setup-node@v4`, runs `npm ci && npm test`.

## Phase 1.1 — Cross-hub call routing

### Goal

Hub A receives a chat request for an AI registered on hub B. Hub A proxies through to B via HTTP, returns the response. Real federation — not just `/registry/global` discovery.

### Approach

When `POST /<name>/v1/chat/completions` arrives at Hub A:

1. Look up `<name>` locally. If found → existing local path.
2. Not found locally → check peer registry. If a peer hub has `<name>`, proxy.
3. Build the proxy request: forward the body + `Authorization: Bearer <api_key>` to peer's `/<name>/v1/chat/completions`.
4. Stream the response back if `stream:true`, else single-shot.
5. If multiple peers claim the name, prefer the local match (already covered) → first peer in `peers` list otherwise.
6. Origin annotation: include `X-Zhub-Origin: <peer_url>` header on the response so callers can see where it came from.

### Auth

The user's `api_key` is the publisher's key on hub B. Hub A doesn't know the key — it just forwards what the client sent. Hub B validates locally. **Hub A never sees the bearer key as anything but an opaque blob.**

### Federation gotchas

- **Loop prevention:** if hub A proxies to hub B which then tries to proxy back to hub A, infinite loop. Mitigation: `X-Zhub-Forwarded-By: <hub-id>` header. If hub sees its own id in this header, refuse with 508.
- **Latency:** every hop = network round-trip. Document the cost.
- **Streaming:** SSE bytes flow through hub A → hub B verbatim. Use `httpx.AsyncClient.stream()` for proxying; pump bytes one chunk at a time.

### Out of scope

- WebSocket connections from a connect()-side client to a federated AI (that requires either WS-tunneling or moving the connection state across hubs — Phase 1.1b).
- Auth-server-mediated federation (signed peer relationships) — Phase 1.2 of the federation track, not this batch.

### TDD

`tests/test_cross_hub_routing.py`:
1. Two hubs in-process (A, B). A peers B.
2. Publish AI `bob` on B with chat handler that returns `"from-b"`.
3. POST to A: `/bob/v1/chat/completions` → expect `"from-b"` response (proxied).
4. POST to A with stream:true → expect SSE chunks arriving from B through A.
5. Loop case: configure A to peer B, B to peer A, publish only on B. Calling A → A forwards to B → B serves locally (no loop). Then publish nothing on either, call A → 404 with peer-checked-but-not-found.

## Phase 1.8 — Tool calls (function calling) end-to-end

### Goal

Standard OpenAI function-calling works through zhub. The AI's `tool_calls` in a chat response automatically map to invoke-request envelopes against connected clients' capabilities. Tool result returns to the AI as a `role: tool` message.

### Why this composes naturally

zhub already has bidirectional invoke. OpenAI tool calls are exactly: "AI wants the runtime to call tool X with args Y, then continue." That maps to: "AI emits invoke-request through hub, gets back invoke-result, continues conversation."

### Architecture

```
[client app]
   POST /<name>/v1/chat/completions
   body has tool definitions (or capabilities discovered from connected client manifest)
                     │
                     ▼
[hub] forwards chat-request to publisher with tools array
                     │
                     ▼
[AI publisher] returns chat-response with tool_calls (instead of plain content)
                     │
                     ▼
[hub] sees tool_calls in publisher's response
   - if tool name matches a connected client's capability:
       → automatic: invoke-request to that client
       → wait for invoke-result
       → append role=tool message
       → forward back to publisher (recursive chat)
   - else: return tool_calls to client app as normal OpenAI shape
                     │
                     ▼
[publisher] continues with tool result in conversation, returns final assistant message
```

### Wire format

Capability declarations from connect()-side already include the JSON schema. Hub turns each capability into a `tool` entry:

```json
{
  "type": "function",
  "function": {
    "name": "send_whatsapp",
    "description": "Send WhatsApp via Loki",
    "parameters": {...JSON schema...}
  }
}
```

These get injected into the chat-request envelope so the AI sees them. AI replies with `tool_calls`; hub maps each to the right connection_id and invokes.

### Operator opt-out

By default: hub auto-resolves tool_calls that match capabilities (saves operator a round-trip). Opt-out: client request adds `X-Zhub-Tool-Resolve: client` header → hub returns tool_calls verbatim and the operator handles them.

### Out of scope

- Parallel tool calls (OpenAI now supports multiple tool_calls per response — phase 1.8b)
- Tool streaming (chunks of tool_calls arriving via SSE) — phase 1.8c
- Tool retries on transient invoke errors — phase 1.8d

### TDD

`tests/test_tool_calls.py`:
1. Publish AI whose chat handler returns a chat-response with `tool_calls: [{name: "send_whatsapp", args: {to: "Ammi", message: "ok"}}]` on first call, then a normal text response on second call (mock the LLM with a deterministic two-step handler).
2. Connect a client exposing `send_whatsapp` capability (returns `{ok: true, delivered: true}`).
3. POST to `/<name>/v1/chat/completions`.
4. Expect: hub auto-resolves the tool_call, appends role=tool message with the result, sends back to publisher, returns publisher's final text response — **client app sees normal chat completion with no tool_calls in the visible response**, but a `tool_results` log in the response usage block.
5. Same flow with `X-Zhub-Tool-Resolve: client` → expect tool_calls returned verbatim, no automatic invoke.
