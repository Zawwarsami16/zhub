# Phase 9.0 — MCP resources + prompts surface

**Date:** 2026-05-10
**Status:** spec
**Goal:** extend `zhub.mcp_server` from tools-only to the full MCP triple — tools + **resources** + **prompts**. After this phase, Claude Desktop / Cursor / Cline pointed at `python -m zhub.mcp_server` see not just `chat` and connected capabilities as tools, but also any operator-declared resources (file-like data) and prompts (parameterized message templates).

---

## Why

The MCP spec defines three primitive surfaces:

- **`tools/list`, `tools/call`** — what we have today (Phase 2.1)
- **`resources/list`, `resources/read`** — file-like data (a markdown doc, a config snapshot, a query result)
- **`prompts/list`, `prompts/get`** — pre-canned message templates with parameter substitution

Hosts use all three. A typical Claude Desktop session might:
- Call a tool to fetch live state
- Read a resource for static reference docs
- Use a prompt template to seed a new conversation with project conventions

zhub's mcp_server only forwards tools. After 9.0 it forwards all three.

## Scope

**In:**
- `Manifest` gains `resources: list[ResourceDecl]` and `prompts: list[PromptDecl]` fields
- `publish()` accepts `resources=[...]` and `prompts=[...]` keyword args
- Hub serves them via existing `/<ai>/manifest.json` (no new endpoint needed)
- `zhub.mcp_server` implements `resources/list`, `resources/read`, `prompts/list`, `prompts/get`
- Static-only for v1: resources carry their content inline; prompts carry their messages inline with `{var}` substitution

**Out (v1):**
- Dynamic resources (publisher round-trip per read)
- Dynamic prompts (publisher round-trip per get)
- Resource subscriptions (`resources/subscribe`, `notifications/resources/updated`)
- Resource templates (URI template parameter substitution server-side)
- Prompt completion / argument validation beyond required-field check

These are all natural Phase 9.x additions if the v1 surface gets used.

## Wire shapes

### Manifest extension

Today's manifest:
```json
{
  "name": "my-ai",
  "description": "...",
  "capabilities": [...],
  ...
}
```

After 9.0:
```json
{
  "name": "my-ai",
  "description": "...",
  "capabilities": [...],
  "resources": [
    {
      "uri": "zhub://my-ai/readme",
      "name": "readme",
      "description": "Project README",
      "mimeType": "text/markdown",
      "content": "# Project\n\n..."
    }
  ],
  "prompts": [
    {
      "name": "summarize",
      "description": "Summarize the given text in 3 bullets",
      "arguments": [
        {"name": "text", "required": true, "description": "The text to summarize"}
      ],
      "messages": [
        {"role": "user", "content": "Summarize this in 3 bullets:\n\n{text}"}
      ]
    }
  ]
}
```

### MCP wire (server-side dispatch in `zhub.mcp_server`)

```
resources/list → {resources: [{uri, name, description?, mimeType?}]}
resources/read {uri} → {contents: [{uri, mimeType?, text}]}
prompts/list → {prompts: [{name, description?, arguments?: [{name, description?, required?}]}]}
prompts/get {name, arguments} → {description?, messages: [{role, content: {type:"text", text}}]}
```

`mcp_server` fetches the AI's manifest on each `*/list` call (or caches like it does for capabilities), builds the response from declared `resources` / `prompts`, and serves `read` / `get` from the inline `content` / `messages` after `{var}` substitution.

## Initialize advertisement

`mcp_server.handle("initialize")` returns `capabilities: {tools: {}}` today. After 9.0:

```json
{
  "capabilities": {
    "tools": {},
    "resources": {},
    "prompts": {}
  }
}
```

Always declare all three. The lists may be empty (graceful when an AI has no resources/prompts), but the surface is always announced so MCP hosts know to call `*/list`.

## Substitution

Prompt messages use simple `{name}` placeholder substitution against the `arguments` dict. Missing required args → JSON-RPC error with code `JSONRPC_INVALID_PARAMS`. Unknown placeholders are left as literal text (no error — operators may want literal braces).

```python
def _substitute(text: str, args: dict[str, Any]) -> str:
    out = text
    for k, v in args.items():
        out = out.replace("{" + k + "}", str(v))
    return out
```

## Files

- `zhub/manifest.py` — add `resources: list[dict]` and `prompts: list[dict]` fields to `Manifest`, defaulting to `[]`
- `zhub/client.py` — `publish()` gains `resources=` and `prompts=` keyword args, threaded into `chat_only_manifest()`
- `zhub/mcp_server.py` — handle `resources/list`, `resources/read`, `prompts/list`, `prompts/get`; advertise capabilities on `initialize`
- `tests/test_mcp_server_resources_prompts.py`
- `zhub/entity.md` — short note in `## routes` about the new surface (no new HTTP route added — MCP-side only)

Estimated ~150 LOC + 4-5 tests.

## Tests

`tests/test_mcp_server_resources_prompts.py`:

1. `initialize_advertises_all_three_surfaces` — capabilities object includes tools, resources, prompts
2. `resources_list_surfaces_manifest_resources` — publish with two resources → mcp_server resources/list returns both
3. `resources_read_returns_inline_content` — read a declared URI → contents match what was declared
4. `resources_read_unknown_uri_errors` — read an unknown URI → JSON-RPC error
5. `prompts_list_surfaces_manifest_prompts` — publish with one prompt → mcp_server prompts/list returns it
6. `prompts_get_substitutes_arguments` — get a prompt with args → messages have `{var}` replaced
7. `prompts_get_missing_required_arg_errors` — missing required arg → JSON-RPC error

## Out of scope (separate phases)

- Dynamic resources where `mcp_server` calls back to the publisher per read
- Resource subscriptions / change notifications
- URI template substitution (`{ai}/readme/{section}` etc)
- Prompt argument JSON-Schema validation beyond required-field check
- `tools/listChanged`, `resources/listChanged`, `prompts/listChanged` notifications
